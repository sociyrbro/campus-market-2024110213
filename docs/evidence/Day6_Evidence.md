## 2. 注册登录设计说明

本项目注册/登录的基本流程如下：
- 注册时将用户数据写入 JSON Server 的 users 资源，包括用户名、密码、显示名称、学院、年级、头像和个人简介；
- 登录时从 users 中查询并校验用户名和密码，验证通过后将当前用户保存到 Pinia 的 userStore 中；
- 同时使用 localStorage 保存当前用户，键名为 'campus-market-current-user'；
- 退出登录时清空 Pinia 中的用户状态和 localStorage；
- 页面刷新时通过 restoreLogin 方法从 localStorage 恢复登录状态，并加载用户的收藏数据。

## 4. 状态持久化说明

为什么需要 localStorage：
- Pinia 中的状态存储在内存中，页面刷新后会丢失；
- localStorage 可以在页面刷新后恢复当前用户信息，保证用户登录状态不丢失；
- App 启动时通过 restoreLogin 方法读取 localStorage 中的用户数据并恢复登录状态；
- 同时在恢复登录时会加载该用户的收藏数据，确保收藏状态也能持久化；
- 本项目只是 Mock 登录，不是生产级安全方案，localStorage 中的用户数据未加密存储。

## 10. 遇到的问题与解决方法

**问题一：收藏功能的类型参数错误**
- 现象：在失物招领页面收藏一个卡片后，转到其他页面相同位置的卡片也显示已收藏；
- 原因：四个业务页面的收藏按钮都错误地使用了 `type: 'trade'`，导致不同业务类型的收藏状态互相干扰；
- 解决方法：将每个页面的收藏类型修改为对应的业务类型（lostFound、errand、groupBuy），通过 `type + id` 组合来唯一标识收藏状态。

**问题二：收藏数据无法按用户保存**
- 现象：登录不同的账号，收藏列表是一样的；
- 原因：favoriteStore 使用全局内存存储收藏数据，所有用户共享同一套收藏列表；
- 解决方法：在 db.json 中添加 favorites 数据集合，每条记录包含 userId；创建收藏相关的 API 接口；修改 favoriteStore 使用 API 操作收藏数据，并在登录时加载用户收藏；修改 userStore 的 login 方法，登录后加载收藏数据。

**问题三：发布页面失物招领类型缺少 publisher 字段**
- 现象：发布失物招领信息后，列表中显示的发布人为空或匿名；
- 原因：createLostFound 调用时没有传递 publisher 字段；
- 解决方法：在发布页面的失物招领类型中添加 `publisher: userStore.displayName`，确保发布人来自当前登录用户。

## 11. 今日反思

通过补充注册/登录功能，我对"用户状态""状态持久化"和"前端 Mock 认证边界"有了更深入的理解。用户状态是前端应用的核心，它决定了用户能看到什么内容、能执行什么操作。状态持久化是保证用户体验的关键，通过 localStorage 可以在页面刷新后恢复用户状态，避免重复登录。前端 Mock 认证边界让我认识到，虽然可以模拟完整的登录流程，但这只是用于学习和演示，真正的生产环境需要后端的安全认证机制，包括密码加密、JWT、权限校验等。同时，状态管理需要考虑数据的隔离性，比如不同用户的收藏数据需要独立存储，避免数据混淆。
