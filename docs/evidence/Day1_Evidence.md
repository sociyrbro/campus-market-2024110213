## 遇到的问题

### 1. nvm 环境变量问题
- **问题描述**：nvm 安装成功后，新终端无法识别 nvm 命令
- **原因分析**：安装程序未正确配置环境变量，或终端未刷新环境变量
- **解决方法**：手动配置 NVM_HOME 和 PATH 环境变量，重启终端后生效

### 2. corepack 权限问题
- **问题描述**：使用 `corepack prepare pnpm@11.9.0 --activate` 时出现权限错误（EPERM）
- **原因分析**：nvm 安装目录权限限制
- **解决方法**：改用 `npm install -g pnpm@11.9.0` 全局安装 pnpm

### 3. 终端环境隔离问题
- **问题描述**：在 AI 沙盒终端中配置的环境变量不会自动应用到用户新打开的终端
- **原因分析**：沙盒终端与用户终端环境隔离
- **解决方法**：重启 VS Code 或手动配置系统环境变量

### 4. tsconfig.app.json 报错问题
- **问题描述**：项目依赖未安装时，TypeScript 无法找到 `@vue/tsconfig/tsconfig.dom.json`
- **原因分析**：缺少 `@vue/tsconfig` 依赖包
- **解决方法**：安装项目依赖后问题解决

## 思考与反思

### 1. 为什么课程没有要求第一天立即开发页面？

**思考结论**：

课程设计的合理性体现在以下几个方面：

#### （1）环境配置是开发的基石
- 没有稳定、统一的开发环境，后续开发会遇到各种配置问题
- Node.js 版本不匹配可能导致项目无法启动
- 包管理器版本差异可能导致依赖安装失败
- 先解决环境问题，避免后续开发中的反复调试

#### （2）理解项目结构是编码的前提
- 不了解项目结构，盲目编码容易把代码放错位置
- 不理解技术栈，可能使用错误的开发方式
- 先理解架构，后续编码才能符合项目规范

#### （3）培养良好的开发习惯
- 第一天建立版本控制意识（Git）
- 第一天理解团队协作的基础（统一环境）
- 第一天学会使用工具（AI 辅助、文档记录）
- 这些习惯比写出几行代码更重要

#### （4）降低学习挫败感
- 如果第一天就开发页面，可能因为环境问题频繁报错
- 先把基础设施搞定，后续学习更顺畅
- 循序渐进的学习路径更符合认知规律

---

### 2. 软件工程中的"理解项目"为什么比"开始编码"更重要？

**思考结论**：

#### （1）盲目编码的风险
- **理解偏差**：不理解需求就编码，可能实现错误的功能
- **设计缺陷**：不了解架构就编码，可能破坏项目结构
- **重复劳动**：不了解已有代码就编码，可能重复实现已存在的功能
- **维护困难**：不理解项目就编码，后期难以维护和扩展

#### （2）理解项目带来的价值
| 方面 | 不理解就编码 | 先理解再编码 |
|------|------------|------------|
| 开发效率 | 反复调试、频繁重构 | 一次实现、减少返工 |
| 代码质量 | 结构混乱、难以维护 | 符合规范、易于扩展 |
| 团队协作 | 破坏他人代码、冲突频发 | 遵守约定、协同顺畅 |
| 学习成长 | 机械复制、不知原理 | 理解本质、举一反三 |

#### （3）软件工程的核心原则
- **先设计后编码**：瀑布模型、敏捷开发都强调设计先行
- **理解需求**：需求分析是软件工程的第一阶段
- **理解架构**：架构设计决定了代码的组织方式
- **理解约束**：技术栈、版本、规范都是编码必须遵守的约束

#### （4）个人体会
今天如果直接开始写代码，可能会：
- 不知道代码应该放在哪个目录
- 不理解为什么用 TypeScript 而不是 JavaScript
- 不理解 Pinia 和 Vue Router 的作用
- 出现环境问题时无从下手排查

先花时间理解项目，后续开发才能"有的放矢"。

### 3. AI 在软件开发中应该承担什么角色？

**思考结论**：

#### （1）AI 的定位：辅助工具，而非主导者

| 角色 | AI 应该做的 | AI 不应该做的 |
|------|------------|--------------|
| 指导者 | 提供建议、方案、解释 | 替代开发者做决策 |
| 加速器 | 加速学习、理解、排查 | 替代开发者的思考过程 |
| 协作者 | 帮助解决常见问题 | 承担核心逻辑设计 |
| 记录者 | 帮助生成文档、总结 | 替代开发者理解内容 |

#### （2）AI 的能力边界
- **擅长**：
  - 快速检索和分析代码
  - 提供常见问题的解决方案
  - 解释技术概念和原理
  - 生成标准化文档和代码片段

- **不擅长**：
  - 理解业务需求和上下文
  - 设计复杂的系统架构
  - 判断方案的适用性
  - 承担代码质量责任

#### （3）开发者应该保持的能力
- **判断能力**：验证 AI 提供的方案是否正确、是否适用
- **理解能力**：理解 AI 生成的代码，而非盲目复制
- **创新能力**：AI 可以提供思路，创新仍需人类
- **责任意识**：代码质量由开发者负责，而非 AI

#### （4）今天的实践体会
- AI 帮助我快速配置环境，但环境变量问题仍需我手动解决
- AI 帮助我分析项目结构，但我需要自己消化理解
- AI 帮助我生成文档，但我需要补充个人思考

**结论**：AI 是强大的助手，但开发者仍是项目的负责人。

### 4. 今天最大的收获是什么？

**思考结论**：

#### （1）不是"跑起来项目"，而是"理解为什么能跑起来"

如果只是按照教程一步步操作，项目跑起来了，但这只是表面成果。

今天的真正收获是理解了：
- 为什么需要 Node.js 24.x 版本（项目约束）
- 为什么使用 pnpm 而非 npm（包管理器效率）
- 为什么项目能通过 TypeScript 检查（配置正确）
- 为什么开发服务器能启动（依赖完整）

#### （2）不是"解决了问题"，而是"学会了如何解决问题"

今天遇到的环境变量问题、权限问题，不仅是解决了，更重要的是理解了：
- 问题产生的原因
- 排查问题的思路
- 解决问题的方法
- 预防问题的措施

这种问题解决能力比单纯完成任务更有价值。

#### （3）不是"使用了 AI"，而是"理解了 AI 的角色"

通过今天的实践，我理解了：
- AI 可以快速提供方案，但需要验证适用性
- AI 可以解释代码，但需要自己消化理解
- AI 可以生成文档，但需要补充个人思考
- AI 是工具，而非替代者

#### （4）建立的核心认知：**"先理解再行动"**

这是今天最大的收获，也是最深刻的反思：

| 传统认知 | 新认知 |
|---------|-------|
| 先编码再调试 | 先理解再编码 |
| 遇到问题再解决 | 理解原理预防问题 |
| 完成任务即可 | 理解为什么完成 |
| 工具是辅助 | 工具需要正确使用 |

这种认知将影响后续所有学习：先理解技术原理，再动手实践；先理解项目架构，再编写代码。
